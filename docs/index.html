Architecture

Three patterns. Infinite extensibility.

Understanding the BASH_SOURCE guard, the router, and uk_common.sh is enough to add a new tool in minutes.

UtilityKit/
├── main.sh            ← router + dashboard
├── setup.sh           ← installer
├── lib/
│   └── uk_common.sh   ← shared helpers
├── _<tool>/
│   ├── _<tool>.sh     ← BASH_SOURCE guarded
│   └── _<tool>_README.md
├── _cache_clean/
│   └── plugins/       ← 17 auto-loaded plugins
├── docs/
├── tests/
│   └── smoke_test.sh
└── CHANGES.md
1
BASH_SOURCE guard
Each tool wraps its entry point in <prefix>_main() and guards top-level execution with [[ "${BASH_SOURCE[0]}" == "${0}" ]]. This lets main.sh safely source all 51 scripts without side effects on startup.
2
Lazy router in main.sh
UK_TOOL_PATHS maps keys to script paths. uk_load <key> sources a tool the first time it is needed (idempotent via UK_TOOL_LOADED). Each route calls <prefix>_main "$@" for scripting or run_<tool>_wizard for the interactive dashboard.
3
Shared visuals: uk_common.sh
All tools source lib/uk_common.sh for colors, icons, prompts, progress bars, and platform detection. Uses a load-once guard (UK_COMMON_SH_LOADED) so nested sourcing is free. NO_COLOR=1 and NO_UNICODE=1 are respected globally.
4
Namespaced functions
Every tool defines a short prefix for all its functions (pi_ for port_inspector, mc_ for media_convert, etc.). Register trap handlers inside <prefix>_main, never at the top level — top-level traps overwrite the parent shell's handlers when sourced.

Plugin System

Extend cache_clean in one file

Drop a script in _cache_clean/plugins/ that implements five functions. The plugin is auto-loaded when its package manager is detected.

17 built-in plugins

npmyarnpnpmbun pipcargogogem composerdotnetconanvcpkg aptpacmandnfbrewapk

Each plugin implements a five-function contract:

<id>_plugin_infoReturns id|display_name|icon
<id>_detectReturns 0 if the package manager is present
<id>_get_cache_dirsPrints cache directory paths, one per line
<id>_scan_cacheEmits orphan entries using cc_emit_* helpers
<id>_clean_orphansCalls cc_clean_orphans_from_file "$1"
plugins/mybuild.sh 
# plugins/mybuild.sh

mybuild_plugin_info() {
  printf 'mybuild|MyBuild|🔨\n'
}

mybuild_detect() {
  command -v mybuild >/dev/null 2>&1
}

mybuild_get_cache_dirs() {
  printf '%s\n' "$HOME/.mybuild/cache"
}

mybuild_scan_cache() {
  local dir
  while IFS= read -r dir; do
    [ -z "$dir" ] && continue
    cc_emit_tot "$dir" "$(cc_du_kb "$dir")"
    while IFS= read -r f; do
      cc_emit_orphan "$dir" "$f" "stale"
    done < <(cc_find_old "$dir" "$CC_OLDER_THAN")
  done < <(mybuild_get_cache_dirs)
}

mybuild_clean_orphans() {
  cc_clean_orphans_from_file "$1"
}
Contributing

Add a new tool in 5 steps

The wiring is spread across several files — edit all five to fully integrate a new tool.

1
Create the tool script
Add _<tool>/_<tool>.sh with namespaced functions and a BASH_SOURCE guard. All function names must share a unique prefix (e.g. xt_ for _xtool). Register traps inside <prefix>_main(), never at the top level.
2
Write a README
Create _<tool>/_<tool>_README.md documenting standalone usage, flags, and examples for users who call the script directly without main.sh.
3
Wire up main.sh
Add an entry to UK_TOOL_PATHS, a case branch routing the CLI command, a run_<tool>_wizard() for the interactive dashboard, and a menu entry on the appropriate "More tools" page.
4
Add smoke tests
Append the CLI command to the cmds=(...) array in tests/smoke_test.sh so <cmd> --help is exercised. Add a behavioral smoke case if the tool has side effects.
5
Use the shared library
Source lib/uk_common.sh and use uk_success, uk_warn, uk_error, uk_confirm, uk_bar, etc. Never re-implement colors, icons, or platform detection. Respect NO_COLOR=1 and NO_UNICODE=1.
✓
Commit style
Use semantic prefixes scoped to the tool:
feat(tool): fix(tool): refactor(tool): docs:
